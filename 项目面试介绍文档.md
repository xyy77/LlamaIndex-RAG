### 1.3 每一步代码实现详解

#### 步骤 1：模型层 — 多模型统一工厂 (`utils.py:44-72`)

```python
def get_llm(selected_label):
    config = ALL_MODELS.get(selected_label)
    provider = config["provider"]
    model_id = config["model_id"]

    if provider == "openrouter":
        return OpenRouter(
            model=model_id,
            api_key=os.getenv("OPENROUTER_API_KEY"),
            temperature=0,        # 设为 0 保证回答确定性，适合知识问答场景
            max_tokens=2048,
            context_window=ctx_window,
        )
    elif provider == "dashscope":
        return OpenAILike(
            model=model_id,
            api_base="https://dashscope.aliyuncs.com/compatible-mode/v1",
            api_key=os.getenv("DASHSCOPE_API_KEY"),
            is_chat_model=True,
            temperature=0,
            max_tokens=2048,
        )
```

**设计要点**：
- 阿里云 DashScope 使用 `OpenAILike` 适配器（因 DashScope 提供 OpenAI 兼容接口）
- OpenRouter 使用官方 `OpenRouter` 集成，自动处理 API 路由
- 所有模型的 `temperature=0`，保证事实类 QA 场景的回答一致性
- `ALL_MODELS` 字典（`utils.py:18-41`）统一管理 4 个模型的 provider、model_id、context_window

#### 步骤 2：文档处理 & 分块层 (`utils.py:94-98`)

```python
# 方案 A：语义分块（适合结构复杂的文档）
if enable_semantic_splitter:
    splitter = SemanticSplitterNodeParser(
        buffer_size=1,
        breakpoint_percentile_threshold=95,
        embed_model=embed_model
    )
# 方案 B：固定大小分块（默认，适合多数场景）
else:
    splitter = SentenceSplitter(chunk_size=1024, chunk_overlap=150)
```

**设计要点**：
- `SentenceSplitter`：chunk_size=1024 token，overlap=150 token，保证上下文连续性
- `SemanticSplitterNodeParser`：基于嵌入相似度在语义边界处断点，breakpoint_percentile_threshold=95 表示在相似度低于 95 分位时断开
- 两种策略通过 `enable_semantic_splitter` 参数切换

#### 步骤 3：双索引构建 (`utils.py:101-113`)

```python
nodes = splitter.get_nodes_from_documents(docs)

# 向量索引 — 用于精确检索细节、引用、事实
summary_index = SummaryIndex(nodes)
vector_index = VectorStoreIndex(nodes)

# 各自创建 Retriever，并包装为 RetrieverTool
summary_retriever = summary_index.as_retriever()
vector_retriever = vector_index.as_retriever(similarity_top_k=15)

summary_tool = RetrieverTool.from_defaults(
    retriever=summary_retriever,
    description="Useful for summarization questions and getting a high-level overview."
)
vector_tool = RetrieverTool.from_defaults(
    retriever=vector_retriever,
    description="Useful for retrieving specific context, details, quotes, or facts."
)
```

**设计要点**：
- **VectorStoreIndex**：将文本块嵌入为向量，做语义相似度检索，适合 "作者的 IBM 1401 是什么机器？" 这类细节问题
- **SummaryIndex**：对全部节点做摘要索引，适合 "这篇文章主要讲了什么？" 这类总览性问题
- 每个 Retriever 包装为 `RetrieverTool`，并赋予自然语言 description，供 LLM 做路由决策

#### 步骤 4：智能路由检索 (`utils.py:117-124`)

```python
# 自定义路由 Prompt，强制输出标准 JSON 双引号格式
new_tmpl = (DEFAULT_SINGLE_SELECT_PROMPT_TMPL +
    "\nCRITICAL: You must use double quotes for all JSON keys and "
    "string values. Do not use single quotes or unquoted keys.")

router_retriever = RouterRetriever(
    selector=LLMSingleSelector.from_defaults(prompt_template_str=new_tmpl),
    retriever_tools=[summary_tool, vector_tool],
    verbose=True
)
```

**设计要点**：
- `LLMSingleSelector`：调用 LLM 判断当前问题应使用哪个检索工具，返回 JSON 格式决策
- **Prompt 优化**：在默认模板后追加 JSON 格式约束，防止模型输出单引号或未加引号的 key 导致解析失败
- Router 输出 `verbose=True`，可在日志中查看路由决策过程

#### 步骤 5：重排序管线 (`utils.py:131-134`)

```python
post_processors = []
if enable_reranker:
    reranker = DashScopeRerank(model="gte-rerank", top_n=5)
    post_processors.append(reranker)
```

**设计要点**：
- 检索阶段取 Top-15 候选节点（recall 优先）
- Reranker 使用 gte-rerank 模型对 15 个候选做精细排序，取 Top-5（precision 优先）
- `enable_reranker=False` 时跳过，在评估脚本中默认关闭以测量基线性能

#### 步骤 6：对话引擎 (`utils.py:136-146`)

```python
memory = ChatMemoryBuffer.from_defaults(token_limit=m_size)

chat_engine = ContextChatEngine.from_defaults(
    retriever=router_retriever,
    memory=memory,
    system_prompt=(
        "你是一个专业的PDF问答助手，要根据提供的文件内容进行回答，"
        "不要编造内容，优先基于提供的PDF内容。"
    ),
    node_postprocessors=post_processors,
    verbose=True
)
```

**设计要点**：
- `ContextChatEngine`：每轮对话都将检索到的上下文注入 LLM，保证回答有据可依
- `ChatMemoryBuffer`：token_limit 设为模型上下文窗口的 50%，既支持多轮对话又避免溢出
- **System Prompt**：强制 "不编造内容，基于 PDF 内容回答"，从 prompt 层面抑制幻觉

## 二、关键技术决策与理由

### 决策 1：为什么选择 LlamaIndex 而不是 LangChain？

**理由**：
- LlamaIndex 专注于数据索引和检索，对 RAG 场景的抽象更自然（Index → Retriever → ChatEngine 的层次清晰）
- 内置 `RouterRetriever` + `LLMSingleSelector`，路由检索可一行代码实现
- 支持丰富的内置 Evaluator（FaithfulnessEvaluator / RelevancyEvaluator）
- 与 DashScope 生态（嵌入、重排序）的集成封装更完整

### 决策 2：为什么设计双索引（Vector + Summary）架构？

**理由**：
- **单索引的局限**：向量检索对所有问题一刀切——问 "全文主旨" 时，Top-K 相似度检索可能只返回局部片段，丢失全局信息
- **分工明确**：SummaryIndex 覆盖 "这篇文章讲了什么？" 等宏观问题，VectorStoreIndex 覆盖 "作者在 9 年级用了什么电脑？" 等细节问题
- **LLM 路由**：通过 `LLMSingleSelector` 让模型自动判断问题类型并选择合适检索器，无需人工规则

### 决策 3：为什么引入 Reranker 重排序？

**理由**：
- 向量检索的 Top-K 召回通常 noise 较多（检索 15 个片段，可能有 5-8 个不相关）
- 重排序模型（gte-rerank）对 query-document 对做精细语义匹配，比向量 cosine 相似度更精准
- **策略**：粗筛（向量检索 Top-15，保证 recall）→ 精排（Reranker Top-5，保证 precision）
- 这是一个经典的 "召回-排序" 两阶段架构

### 决策 4：为什么选择这些模型做对比？

**理由**：
- **Qwen Max**：阿里云旗舰模型，中文能力强，作为商业模型的代表
- **Mistral Small 3 (24B)**：中等规模开源模型中性价比最高的之一，验证 "小模型 + RAG" 可行性
- **Llama 3.1 8B**：当前最流行的开源 8B 模型，验证极小模型在 RAG 场景的边界能力
- **Qwen3.6 Plus**：对比 Qwen 系列的升级效果
- 覆盖了商业 vs 开源、大 vs 中 vs 小规模的完整矩阵

### 决策 5：为什么选择 Faithfulness（忠实度）和 Relevancy（相关性）作为评估指标？

**理由**：
- **Faithfulness（忠实度）**：RAG 的核心价值是 "基于文档回答"，忠实地衡量回答是否来自检索到的上下文、是否存在幻觉编造——这是 RAG 区别于纯 LLM 的关键指标
- **Relevancy（相关性）**：衡量回答是否切题，检索是否召回了正确的内容——直接反映检索管线质量
- 这两个指标分别从 "内容可信度" 和 "检索质量" 两个维度评估 RAG 系统

### 决策 6：为什么用独立评估模型（Qwen Max）而不是被评估模型自评？

**理由**：
- 避免 "自己评自己" 的 bias：Llama 3.1 8B 评判自己的回答时可能过于宽松
- 用更强的模型（Qwen Max）做评估，类似 "老师评学生" 的模式，评估结果更客观公正
- 这是 RAG 评估领域的最佳实践（LLM-as-Judge with stronger evaluator model）

### 决策 7：为什么默认关闭 Reranker 做评估？

**理由**：
- 评估目标是测量 **模型理解能力 + 检索质量** 的原始组合
- 加入 Reranker 会掩盖检索器的原始质量差异
- 在 `eval.py` 中 `enable_reranker=False`，使得评估结果反映的是模型的底线能力

### 决策 8：为什么选择 Streamlit 而不是 Gradio 或其他框架？

**理由**：
- Streamlit 的 Python 原生编程模型最简单，适合快速原型验证
- 支持 session_state 管理多轮对话状态
- 内置 `st.chat_message` / `st.chat_input` 聊天组件，无需写前端代码
- 支持 `st.empty()` + 逐帧更新实现流式输出效果

## 三、可量化的成果

### 3.1 评估数据集

- 基于 Paul Graham 的《What I Worked On》英文文章，人工标注 **44 条** QA 对
- 每条包含：`query`（问题）、`reference_answer`（参考答案）、`reference_contexts`（原文出处）
- 问题类型覆盖：事实提取、人物关系、时间线、因果关系、观点理解等

### 3.2 模型性能对比（SentenceSplitter 分块，无 Reranker）

| 模型 | Faithfulness（忠实度） | Relevancy（相关性） |
|------|----------------------|-------------------|
| **Mistral Small 3 (24B)** | **95.45%** (42/44) | **90.91%** (40/44) |
| Llama 3.1 8B | 79.55% (35/44) | 84.09% (37/44) |

### 3.3 语义分块 vs 固定分块对比

| 模型 | 分块策略 | Faithfulness | Relevancy |
|------|---------|-------------|-----------|
| Llama 3.1 8B | SentenceSplitter | 79.55% | **84.09%** |
| Llama 3.1 8B | SemanticSplitter | **81.82%** | 70.45% |
| Mistral Small 3 | SentenceSplitter | **95.45%** | **90.91%** |
| Mistral Small 3 | SemanticSplitter | 90.91% | 88.64% |

### 3.4 关键发现

1. **Mistral Small 3 是最优选择**：24B 中等规模模型在 RAG 场景下忠实度达到 95.45%，甚至超过更大模型的预期表现，说明模型对检索上下文的利用效率很高
2. **语义分块对忠实度有提升**：Llama 3.1 8B 用语义分块后忠实度从 79.55% 提升到 81.82%
3. **语义分块可能降低相关性**：两个模型在语义分块下相关性均略降（84.09%→70.45%, 90.91%→88.64%），可能是语义切分导致部分信息片段被截断
4. **8B 小模型的 RAG 边界**：Llama 3.1 8B 在 RAG 场景下忠实度不足 80%，说明 8B 模型在理解英文长文上下文时仍有明显局限

## 四、STAR 法则简历项目介绍（适用于 AI 产品经理实习）

### 项目名称：RAG Chat — 基于多模型对比的 PDF 智能问答系统

---

**S — Situation（项目背景与痛点）**

企业内部知识库（PDF 文档、合同、论文等）存在信息检索效率低下的痛点：传统关键词搜索无法理解语义，而直接使用通用大模型（如 ChatGPT）无法基于特定文档回答，容易产生幻觉编造。市面上缺乏一个支持多模型灵活切换、有完整评估体系的 RAG 解决方案。

---

**T — Task（目标与我的角色）**

**目标**：从 0 到 1 构建一个支持多模型的 RAG 智能问答系统，横向评估不同模型的 RAG 效果，为 AI 产品选型提供数据支撑。

**我的角色**：负责需求分析、技术方案设计、模型选型策略、评估指标体系制定，以及产品效果量化评测。

**核心目标拆解**：
1. 定义产品核心用户场景（论文精读、合同审查、技术文档问答）
2. 设计可扩展的多模型接入架构
3. 建立 RAG 质量评估体系（指标选择 + 数据集构建）
4. 完成跨模型对比评测，输出选型建议

---

**A — Action（具体行动，突出产品思维）**

**1. 需求分析与场景定义**
- 梳理了 3 个核心使用场景：论文阅读（需要细节定位）、合同审查（要求高忠实度）、技术手册查询（需全局理解 + 细节检索）
- 基于场景分析，确定了 "双索引架构" 的产品需求：向量检索定位细节 + 摘要检索把握全局

**2. 模型选型策略**
- 制定了 "商业 vs 开源 × 大规模 vs 小规模" 的 2×2 模型对比矩阵
- 纳入了 Qwen Max（商业）、Mistral Small 3（开源中规模）、Llama 3.1 8B（开源小规模）、Qwen3.6 Plus（新代模型）共 4 个模型

## 五、简历项目介绍（优化版）

### 5.1 项目背景与目标

**痛点**：长文档（PDF 合同、论文、技术手册）场景下，传统关键词搜索无法理解语义，用户定位信息效率低；直接使用通用大模型（如 ChatGPT）回答文档问题时，模型无法基于特定文档内容回答，容易产生幻觉/编造信息。

**目标**：从 0 到 1 构建端到端 RAG（检索增强生成）系统，实现"上传 PDF → 自然语言提问 → 基于文档内容可靠回答"的闭环。同时建立可量化的评估体系，横向对比不同 LLM 和分块策略在 RAG 场景下的效果差异，输出模型选型依据。

---

### 5.2 项目方案

#### 整体架构

```
用户上传 PDF/TXT → 文档解析(SimpleDirectoryReader) → 文本分块 → 双索引构建
→ 智能路由检索(RouterRetriever + LLMSingleSelector) → 重排序(gte-rerank)
→ 对话生成(ContextChatEngine + ChatMemoryBuffer) → 流式输出(Streamlit)
```

**技术栈**：LlamaIndex（RAG 编排） + Streamlit（UI） + DashScope/OpenRouter（模型服务）

#### 方案细节（对应代码）

**① 文档加载与分块**（`utils.py:94-99`，`demo.py:114-117`）

用户上传文件后，写入临时目录 `tempfile.mkdtemp()`，通过 `SimpleDirectoryReader` 解析 PDF/TXT。支持两种分块策略可切换（`enable_semantic_splitter` 参数）：
- **SentenceSplitter**（默认）：`chunk_size=1024, chunk_overlap=150`，固定 token 数切分，重叠保证上下文连续性
- **SemanticSplitterNodeParser**：`buffer_size=1, breakpoint_percentile_threshold=95`，基于 DashScope text-embedding-v2 的嵌入相似度在语义边界处动态断点

**② 双索引 + 智能路由检索**（`utils.py:101-124`）

对切分后的节点同时构建两种索引：
- **VectorStoreIndex**：生成向量嵌入（DashScope text-embedding-v2），支持语义相似度检索，定位细节、引用、事实
- **SummaryIndex**：文档摘要索引，覆盖概括性、整体理解类问题

两种索引各自封装为 `RetrieverTool`，赋予自然语言描述（`"Useful for summarization questions..."` / `"Useful for retrieving specific context..."`），注入 `RouterRetriever`。路由决策由 `LLMSingleSelector` 完成——让 LLM 读取问题后判断该用哪个检索器，以 JSON 格式输出选择结果。为防止 JSON 格式错误，自定义 Prompt 模板追加了强制双引号约束（`utils.py:118`）：
```python
new_tmpl = DEFAULT_SINGLE_SELECT_PROMPT_TMPL + "\nCRITICAL: You must use double quotes..."
```

**③ 重排序管线**（`utils.py:131-134`）

检索阶段 `VectorStoreIndex` 召回 Top-15（保证 recall），经由阿里云 `gte-rerank` 模型对 15 个候选做精细语义匹配，输出 Top-5（保证 precision）。通过 `DashScopeRerank(model="gte-rerank", top_n=5)` 实现，作为 `ContextChatEngine` 的 `node_postprocessors` 注入。重排序可通过 `enable_reranker` 参数开关。

**④ 对话引擎**（`utils.py:126-147`）

使用 `ContextChatEngine` 将检索到的上下文注入 LLM，配合自定义 System Prompt（`"你是一个专业的PDF问答助手，不要编造内容"`）从 Prompt 层抑制幻觉。`ChatMemoryBuffer` 的 token_limit 设为模型上下文窗口的 50%，支持多轮对话。所有模型 `temperature=0` 以保证事实类回答的一致性。

**⑤ 多模型统一接入**（`utils.py:18-72`）

通过 `ALL_MODELS` 配置字典 + `get_llm()` 工厂函数，统一接入 DashScope 和 OpenRouter 两套 API：

| 模型 | Provider | API 适配方式 |
|------|---------|-------------|
| Qwen Max | DashScope | `OpenAILike`（DashScope 兼容 OpenAI 接口） |
| Mistral Small 3 (24B) | OpenRouter | `OpenRouter` 官方集成 |
| Llama 3.1 8B | OpenRouter | `OpenRouter` 官方集成 |
| Qwen3.6 Plus | OpenRouter | `OpenRouter` 官方集成 |

**⑥ 前端交互**（`demo.py`）

- 用 Streamlit 的 `st.chat_message` / `st.chat_input` 构建聊天界面
- `stream_chat()` 实现 token 级流式输出，`st.empty()` 占位 + 逐 token 更新 + `▌` 光标动画
- 正则匹配 `<think>...</think>` 标签，用 `st.expander` 折叠展示模型推理过程
